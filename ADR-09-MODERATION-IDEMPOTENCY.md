# ADR-09: Идемпотентность обработки Moderation событий

## Статус
Принято

## Контекст
Необходимо обеспечить идемпотентную обработку событий от сервиса Moderation для применения решений по блокировке/одобрению товаров. Повторная отправка одного и того же события не должна изменять состояние товара или создавать дубликаты записей в системе.

## Вариант 1: Таблица `ProcessedModerationEvent`
- **Описание**: Ведется журнал обработанных событий по (service_id, idempotency_key) с уникальным индексом
- **Преимущества**: 
  - Простая реализация
  - Не требует изменений основной модели Product
  - Явная история всех обработанных событий (audit trail)
- **Недостатки**:
  - Требует дополнительной таблицы в БД
  - Race condition между проверкой и вставкой (решается SELECT FOR UPDATE)
  - Может вырасти очень большой

## Вариант 2: Поле в модели Product
- **Описание**: Добавить поля `last_moderation_event_id` и `last_moderation_service_id` в Product
- **Преимущества**:
  - Не требует отдельной таблицы
  - Быстрая проверка (один запрос)
- **Недостатки**:
  - Миграция требуется для каждого товара
  - Потеря истории предыдущих событий
  - Требует миграции существующих данных

## Вариант 3: Upsert с условием
- **Описание**: Использовать UPDATE ... WHERE ... EXISTS для атомарного обновления
- **Преимущества**:
  - Один SQL-запрос вместо двух
  - Нет явной проверки существования
- **Недостатки**:
  - Сложнее в реализации и тестировании
  - Django ORM не имеет встроенной поддержки для сложных условий
  - Требует сырых SQL-запросов

## Решение
**Выбран Вариант 1: Таблица `ProcessedModerationEvent`**

### Обоснование по критериям:
1. **Race condition**: SELECT FOR UPDATE блокирует строку товара, гарантируя атомарность
2. **Сложность поддержки**: Минимальна — простой уникальный индекс, явная семантика
3. **Масштабируемость**: ProcessedModerationEvent можно архивировать по дате, а Product остается чистой
4. **Testability**: Легко проверить в тестах через ProcessedModerationEvent.objects.count()

### Техническая реализация:
```python
# В представлении ModerationEventApplyView
with transaction.atomic():
    # Проверка идемпотентности (быстрая)
    if ProcessedModerationEvent.objects.filter(
        service_id=service_id,
        idempotency_key=idempotency_key,
    ).exists():
        return Response(status=204)
    
    # Блокировка строки товара для изоляции
    product = Product.objects.select_for_update().get(id=product_id)
    
    # Применение логики
    apply_moderation_logic(product, event_data)
    product.save()
    
    # Регистрация обработки
    ProcessedModerationEvent.objects.create(
        service_id=service_id,
        idempotency_key=idempotency_key,
    )
```

### Гарантии:
- Уникальный индекс по (service_id, idempotency_key) предотвращает дубликаты
- SELECT FOR UPDATE изолирует изменения товара от параллельных запросов
- Атомарная транзакция гарантирует консистентность БД
